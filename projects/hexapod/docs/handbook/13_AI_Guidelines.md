# 第十三章 AI开发规范

本章是**强制规范**。凡使用 AI 编程助手、代码生成、自动重构、或「让模型直接写一段让机器人动起来的代码」的场景，均必须遵守。违反本章的补丁，无论效果多炫，在本项目中视为不合格实现。

> **一句话**  
> AI 可以写动作策略与 Motion 调用，**不可以**写 PWM / PCA9685 直控，也**不可以**绕过速度、中断、暂停、恢复与安全检查。

---

## 13.1 适用范围与效力

<a id="scope-authority"></a>

### 13.1.1 对谁强制

- 生成新动作实现的 AI  
- 修改步态 / 姿态 / 表演代码的 AI  
- 编写 APP 胶水层、语音意图处理、自动播放器的 AI  
- 编写测试与模拟的 AI（测试可 mock Motion，仍不得引入生产路径上的直控舵机模块）  
- 人类开发者在审查 AI 输出时，同样按本章清单验收  

### 13.1.2 效力级别

| 级别 | 含义 |
|------|------|
| MUST | 必须；违反即打回 |
| MUST NOT | 禁止；违反即打回 |
| SHOULD | 强烈应当；例外需书面理由 |
| MAY | 允许 |

本章下列「红线」全部为 MUST / MUST NOT。

---

## 13.2 红线清单（开篇必读）

<a id="red-lines"></a>

以后所有 AI 写动作必须遵守：

1. **MUST NOT** 直接控制 PWM。  
2. **MUST NOT** 直接控制 PCA9685（或任何舵机驱动芯片的寄存器/通道写法）。  
3. **MUST** 所有动作都调用 **Motion 层**。  
4. **MUST** 所有动作都支持 **速度参数**（`speed`）。  
5. **MUST** 所有动作都支持 **中断**。  
6. **MUST** 所有动作都支持 **暂停**。  
7. **MUST** 所有动作都支持 **恢复**。  
8. **MUST** 做进入 / 退出姿态检查（或显式调用 Motion 提供的检查）。  
9. **MUST** 遵守安全限位。  
10. **MUST** 做重心 / 支撑相关检查（由 Motion 安全服务提供的接口）。  
11. **MUST** 打日志并携带 **MotionID**。  
12. **MUST NOT** 硬编码魔术角度散落在业务代码各处。  
13. **MUST** 实现前阅读本手册对应条目（及第十一章库规范）。  
14. **MUST** 通过测试清单与 Code Review 清单。

## 13.3 架构边界：AI 允许写到哪一层

<a id="architecture-boundary"></a>

```text
[ APP / 语音 / 脚本 / AI编排 ]
              │ 只传 MotionID + params
              ▼
        [ 动作库 catalog ]
              │
              ▼
          [ Motion 层 ]  ← AI 实现动作的主战场
              │ 轨迹、节拍、状态机、暂停恢复
              ▼
        [ 安全与限位服务 ]
              │
              ▼
        [ 关节目标接口 / 驱动适配 ]
              │
              ▼
     （驱动细节对动作作者不可见）
```

### 13.3.1 AI 可以做的事

- 新增 `MOVE_` / `POSTURE_` / `EMO_` … 的 Motion 状态机与轨迹生成。  
- 把手册公式变成「足端轨迹 → 逆解请求 → Motion 执行」。  
- 编写组合动作对子 Motion 的编排。  
- 写单元测试：速度缩放、中断、暂停恢复、错误码。  
- 更新动作 JSON 与 handbook_ref。  

### 13.3.2 AI 禁止做的事

- 打开 I2C/SPI 去拨通道。  
- 计算 `pulse = angle * a + b` 并直接写出。  
- 在 APP 层缓存十八路角度并「自己定时器刷舵机」。  
- 复制粘贴驱动示例进动作文件「先跑起来再说」。  
- 为了通过编译而 `import` 底层驱动模块到动作包。  

### 13.3.3 为什么必须分层

PWM/PCA9685 直控会跳过：速度曲线、关节限位、碰撞包络、重心检查、统一急停、暂停语义、日志关联。短期「能动」等于长期「不可维护且危险」。本手册从根上禁止 AI 走这条捷径。

## 13.4 禁止：直接控制 PWM

<a id="no-pwm"></a>

### 13.4.1 规则

**MUST NOT** 在动作、APP、行为脚本中设置或暗示 PWM 占空比、脉冲宽度、定时器比较值作为控制手段。

### 13.4.2 文字正例

「生成 `MOVE_FORWARD`：在 Motion 步态播放器中注册周期轨迹；接受 `speed`；响应 abort/pause/resume；足端目标交给 Motion 的关节目标接口。」

### 13.4.3 文字反例

「为了让腿抬起来，AI 直接写：把某定时器通道设成某某微秒脉冲，循环刷新。」——**不合格**。即使注释写「临时」，也不允许合入动作路径。

### 13.4.4 常见伪装（同样禁止）

- 把 PWM 叫作 `signal`、`duty`、`compare` 但仍直达硬件。  
- 在「工具脚本」目录写直控，再被动作 `include`。  
- 测试里操作真硬件 PWM 而非 mock Motion。  

---

## 13.5 禁止：直接控制 PCA9685

<a id="no-pca9685"></a>

### 13.5.1 规则

**MUST NOT** 直接读写 PCA9685 或同类多路舵机驱动的通道寄存器、预分频、on/off tick。动作作者不感知「腿 1 的 J2 是通道几」。

### 13.5.2 为什么单独列一条

历史上很多教程以 PCA9685 为中心讲机器人，AI 模型极易复述教程结构。本项目明确：教程结构不等于本仓库结构。通道映射属于驱动适配层，动作层只认腿号 1～6 与关节 J1/J2/J3 的**逻辑目标**（经 Motion）。

### 13.5.3 文字正例

「`LEG_LIFT` 接收 `leg_id`；Motion 查询该腿标定与限位；生成抬腿轨迹；安全服务确认支撑与重心后执行。」

### 13.5.4 文字反例

「根据腿号查一张通道表，对 PCA9685 第 N 通道写入角度换算值。」——**不合格**。

---

## 13.6 必须：调用 Motion 层

<a id="must-use-motion"></a>

### 13.6.1 规则

**MUST** 一切可见行为以 Motion 调度进入：`play` / `abort` / `pause` / `resume` / `queue`（具体 API 名称以工程为准，语义不可缺）。

### 13.6.2 动作实现位置

- 官方动作：Motion 模块内的动作类/脚本 + 第十一章 JSON。  
- 用户组合：仍调用 Motion 播放子 ID。  
- APP 按钮：只发 ID 与参数。  

### 13.6.3 文字正例

「语音『打招呼』→ 解析为 `EMO_WAVE_RIGHT` → `motion.play(id, {speed:0.5, intensity:0.6})`。」

### 13.6.4 文字反例

「语音回调里直接 for 循环改三个关节角并 sleep。」——**不合格**。

---

## 13.7 必须：支持速度参数

<a id="must-speed"></a>

### 13.7.1 规则

**MUST** 每一动作实现尊重 `speed`（或项目统一同义参数），并产生可观察的时间缩放效果。

### 13.7.2 语义

- `speed` 缩放轨迹进度时间，而不是跳过安全检查。  
- 儿童模式上层可夹取 max speed；动作实现仍应正确响应传入值。  
- 不得写死 `sleep(100ms)` 作为唯一节拍却忽略 `speed`。  

### 13.7.3 文字正例

「相位时间 `dt_eff = dt_nom / clamp(speed)`；整段姿态过渡按 speed 缩放。」

### 13.7.4 文字反例

「只有默认一种快慢；参数里有 speed 字段但实现未读取。」——**不合格**。

### 13.7.5 与 intensity 的关系

AI **SHOULD** 同时支持 `intensity` 并保持与 `speed` 正交。不得用 speed 一个旋钮同时硬绑死「又快又猛」而导致儿童限速失效时振幅仍过大——限幅应由安全层与模式策略共同完成。

---

## 13.8 必须：支持中断

<a id="must-interrupt"></a>

### 13.8.1 规则

**MUST** 动作在执行循环中可被中断；急停与安全故障必然可中断。

### 13.8.2 实现要点（语义层）

- 长循环每步检查 abort 标志。  
- 收到 abort：停止推进轨迹，按动作声明进入安全退出（或交由全局安全停机）。  
- 不得 `detach` 出无法取消的底层硬件线程还继续写关节。  

### 13.8.3 文字正例

「步态循环每帧查询 `context.should_abort()`；为真则调用 `safe_exit()` 并返回中断完成状态。」

### 13.8.4 文字反例

「`for i in 1..1000: 走一步; sleep;` 无检查，只能等跑完。」——**不合格**。

### 13.8.5 关于 `interruptible: false`

仅允许极短临界段，且 **急停仍可破**。AI **MUST NOT** 把整段表演标成不可中断以「保证好看」。

---

## 13.9 必须：支持暂停

<a id="must-pause"></a>

### 13.9.1 规则

**MUST** 支持暂停：逻辑时钟停止，机器人保持安全静默姿态或定义的 pause 姿态。

### 13.9.2 文字正例

「收到 pause：冻结步态相位；保持当前足端/关节目标；状态记为 Paused。」

### 13.9.3 文字反例

「暂停=直接 power off 舵机」或「暂停=什么都不做但相位继续走导致恢复时瞬移」——**不合格**。

---

## 13.10 必须：支持恢复

<a id="must-resume"></a>

### 13.10.1 规则

**MUST** 支持从暂停点恢复；默认 `resume_policy = continue`。

### 13.10.2 文字正例

「resume：解除冻结，自相位 p 继续；若环境变化触发安全失败，则拒绝恢复并要求重新 enter 检查。」

### 13.10.3 文字反例

「resume 重新从头播放却不提示」——仅当声明 `restart` 才允许，否则不合格。

---

## 13.11 进入 / 退出姿态检查

<a id="enter-exit-checks"></a>

### 13.11.1 规则

**MUST** 在动作开始前做进入检查，在结束时做退出检查（或显式声明交由 Motion 统一钩子）。

### 13.11.2 进入检查关注什么

- 当前是否已急停锁存。  
- 电池 / 过温等健康位（若系统提供）。  
- 当前姿态是否在该动作允许的起始包络（例如凌空调试腿时不得当作支撑步态启动）。  
- 所需腿号合法。  
- 模式是否允许（儿童/展厅）。  

### 13.11.3 退出检查关注什么

- 是否到达声明的结束姿态或可接受邻域。  
- 是否所有腿离开危险奇异区。  
- 是否把控制权交回待机动作。  

### 13.11.4 文字正例

「`MOVE_FORWARD` 要求从站立包络启动；若当前是 Home 非承重姿态，先自动或要求先 `POSTURE_STAND`。」

### 13.11.5 文字反例

「任意姿态直接开跑，靠舵机硬拧到第一步」——**不合格**。

---

## 13.12 安全限位

<a id="joint-limits"></a>

### 13.12.1 规则

**MUST** 所有关节目标经限位服务裁剪或拒绝；指令行程认知为 0°～180°，软件软限位应更保守，并尊重碰撞文档中的邻腿/触腹约束。

### 13.12.2 Home 仅作参考

Home（90/180/125）是静止参考，不是「可以随便加减 30°」的许可证。AI **MUST NOT** 假设 Home±Δ 一定安全。

### 13.12.3 文字正例

「轨迹采样后批量 `limits.clamp(leg, joint, angle)`；若偏离需求过大则失败返回，而不是静默扭曲成另一动作。」

### 13.12.4 文字反例

「为了抬更高，把 J2 写成 200°」或「知道会撞邻腿但先看效果」——**不合格**。

---

## 13.13 重心与支撑检查

<a id="com-checks"></a>

### 13.13.1 规则

**MUST** 对涉及抬腿与步态的动作使用重心/支撑检查接口（名称以工程为准）。单腿抬起前确认其余支撑充分；三角步态在相位切换时确认支撑三角形策略被遵守。

### 13.13.2 为什么

十八自由度六足在错误相位同时抬三条不相邻腿时，会直接翻倒。AI 若只优化「好看的抬腿高度」而忽略支撑，属于危险生成。

### 13.13.3 文字正例

「`LEG_LIFT`：检查 leg_id 不在唯一支撑集；COM 投影仍在支撑多边形内（或系统简化规则通过）再抬。」

### 13.13.4 文字反例

「六条腿依次快速抬起做波浪，却无任何支撑约束」——若未通过安全模型验证，不得标 `released`。

---

## 13.14 日志与 MotionID

<a id="logging-motionid"></a>

### 13.14.1 规则

**MUST** 在动作生命周期日志中携带 `motion_id`，并建议携带：`request_id`、`speed`、`intensity`、`leg_id`、结果码、中断原因。

### 13.14.2 为什么

没有 MotionID 的日志在展厅排障时等于噪音。AI 生成的 `print("ok")` 不符合规范。

### 13.14.3 文字正例

「`INFO motion_start id=MOVE_FORWARD speed=0.42 intensity=0.5`」  
「`WARN motion_abort id=MOVE_FORWARD reason=estop`」

### 13.14.4 文字反例

「只打 `done`」或打印一堆硬件通道值——后者还可能泄露错误抽象层。

---

## 13.15 禁止魔术角度散落

<a id="no-magic-angles"></a>

### 13.15.1 规则

**MUST NOT** 在业务代码多处硬编码 `90`、`180`、`125`、`135` 等散落数字表达姿态语义。

### 13.15.2 正确做法

- Home 与标称角来自标定/常量模块（单一来源）。  
- 轨迹参数进入动作配置 JSON。  
- 允许在测试中出现明确具名常量。  

### 13.15.3 文字正例

「`angles = calibration.home_pose()`；抬腿高度由 `intensity` 与配置 `lift_span_deg` 计算。」

### 13.15.4 文字反例

「在 APP、语音、动作、工具脚本里各写一套 125、170、45」——**不合格**。

---

## 13.16 必须先读手册对应条目

<a id="read-handbook-first"></a>

### 13.16.1 规则

AI **MUST** 在实现某动作前阅读：

1. 本手册中描述该动作机理的章节条目；  
2. 第十一章：命名、参数、分类、审核；  
3. 本章红线；  
4. 相关安全文档（关节约定、碰撞约定、机体参数）。  

### 13.16.2 输出时应声明

建议在 PR / 提交说明中写：

- 实现的 MotionID  
- 参考的手册锚点  
- 支持的参数列表  
- 已知限制  

### 13.16.3 为什么

未读手册的 AI 会用「四足教程」或「舵机扫角教程」污染六足项目，导致坐标系（+X 前 +Y 左）与腿序错误。

## 13.17 标准实现流程（AI 操作规程）

<a id="implementation-procedure"></a>

1. **定位语义** → 选定前缀与 MotionID（第十一章）。  
2. **阅读条目** → 手册 + 碰撞/关节约束。  
3. **写/更新 JSON** → capabilities 全开：speed/pause/resume/interruptible/检查项。  
4. **实现 Motion 状态机** → 只调 Motion/安全接口。  
5. **自测清单** → 见 13.18。  
6. **请求审查** → 见 13.19。  
7. **入库** → catalog `released` 或先 `sandbox`。  

跳过步骤 2、5、6 直接「生成一大段驱动代码」的行为，视为流程违规。

## 13.18 测试清单

<a id="test-checklist"></a>

AI 提交动作前，至少完成（真机或合格模拟器）：

### 功能

- [ ] 默认参数可完整播放  
- [ ] `speed` 低/中/高可区分  
- [ ] `intensity`（若适用）可区分  
- [ ] `loop=true/false` 行为正确  
- [ ] `leg_id` 非法值被拒绝  

### 控制语义

- [ ] 播放中 abort 立即响应  
- [ ] 急停锁存后不可继续瞎跑  
- [ ] pause 后关节不无失控漂移  
- [ ] resume 从暂停点继续（或声明 restart）  
- [ ] 连点不会破坏安全（风暴策略）  

### 安全

- [ ] 限位不会被目标角突破  
- [ ] 抬腿/步态通过重心与支撑相关检查  
- [ ] 进入非法起始姿态时失败或自动安全过渡  
- [ ] 儿童速度夹取不被动作内部覆写  

### 可观测性

- [ ] 日志含 MotionID 与关键生命周期事件  
- [ ] 失败有明确错误码/原因  

### 回归

- [ ] 不影响 Home 与急停通路  
- [ ] 不引入对 PWM/PCA9685 的依赖

## 13.19 Code Review 清单

<a id="code-review-checklist"></a>

审查者对 AI 变更逐条勾选：

### 红线

- [ ] 无 PWM 直控  
- [ ] 无 PCA9685/通道直控  
- [ ] 仅通过 Motion 层对外表现  
- [ ] 支持 speed / 中断 / 暂停 / 恢复  

### 设计

- [ ] MotionID 符合前缀与命名  
- [ ] JSON 与实现一致  
- [ ] 无散落魔术角度  
- [ ] handbook_ref 有效  
- [ ] 坐标系与腿序正确（+X 前，+Y 左，腿 1～6）  

### 安全

- [ ] 进入/退出检查  
- [ ] 限位  
- [ ] 重心/支撑  
- [ ] 场景标签合理（child/exhibit/adv）  

### 品质

- [ ] 测试已补或已跑  
- [ ] 日志充分  
- [ ] 无「先这样以后再改」的硬件捷径注释蒙混  

**任一项红线未勾，禁止合并。**

## 13.20 正反例合集（长文对照）

<a id="examples-good-bad"></a>

下列均为**文字描述**，不提供可运行驱动代码。

### 例 A：前进步态

**正例**  
实现 `MOVE_FORWARD`：三角步态相位机位于 Motion；`speed` 调步频；`intensity` 调步长/步高；每帧安全检查；可 pause/resume；abort 进入减速停步；catalog 标记 `child_safe` 且 APP 夹速度。

**反例**  
「写一个循环：按通道表依次给十八条腿塞角度，用固定延时。」——无 Motion、无速度语义、无中断。

### 例 B：Home 姿态

**正例**  
`POSTURE_HOME` 读取标定 home；经 Motion 平滑过渡；速度可调；可中断；结束后 exit_check。

**反例**  
「三个即时赋值 90/180/125 写入驱动并 sleep(1s)」——无平滑、无中断、魔术角散落。

### 例 C：抬腿演示

**正例**  
`LEG_LIFT` + `leg_id`；先 com_check；再抬；日志记录腿号；儿童模式隐藏。

**反例**  
「随便抬腿 1，同时允许 UI 再点腿 2、3」导致支撑丧失——缺互斥与检查。

### 例 D：展厅节目

**正例**  
`PERF_OPENING` 内部顺序调用已 released 子动作；总节目可 abort；自动播放器只消费 MotionID。

**反例**  
「节目脚本直接底层扫舵机做灯光式抖动」——绕过库与安全。

### 例 E：语音

**正例**  
词表映射 ID；「急停」走安全通路；低置信度不执行。

**反例**  
「把识别到的数字当脉冲宽度」——严重违规。

### 例 F：暂停恢复

**正例**  
暂停冻相位；恢复续跑；恢复前快速健康检查。

**反例**  
「暂停即断开控制，恢复时按最新目标瞬间拉开大角度」——可能打齿与翻倒。

### 例 G：参数

**正例**  
对外只暴露 speed/intensity/duration/loop/leg_id。

**反例**  
「暴露 channels[18] 与 pulse_us[18] 给 APP」——把用户变成驱动工程师，且必炸。

### 例 H：版本与文档

**正例**  
改默认步高 → MINOR 或 MAJOR（视包络）；更新 CHANGELOG 与手册。

**反例**  
「悄悄改 `MOVE_FORWARD` 轨迹导致展厅站台掉落风险，还不改版本号」。

## 13.22 违规分级与处理

| 级别 | 例 | 处理 |
|------|----|------|
| 致命 | PWM/PCA 直控进主路径 | 立即打回；撤销 released |
| 严重 | 无急停中断、无限位 | 打回；禁止展厅使用 |
| 中等 | 无 pause/resume、无 speed | 打回 |
| 轻微 | 日志缺字段、文案不清 | 可先 sandbox，修后 released |

## 13.20 跨领域实现时的 AI 禁令补丁

当提示词要求「学 Spot」「学蜘蛛」「做个王者荣耀式登场」时，AI 仍必须遵守：

1. **只许生成 Motion 层代码与配置**，不许生成 PCA9685 通道表、PWM 占空比、wiring 调试脚本作为动作实现。
2. **先检索本手册条目**：若无对应 MotionID，先提案 JSON 契约与手册草稿段落，再写实现。
3. **游戏/生物灵感必须本地化到 18DOF 与 MG996R**：禁止假设存在腰关节、脚趾、表情屏幕（除非硬件真实存在）。
4. **危险方言默认关闭**：逃窜、踢击、冲刺、越障必须 `product_packs` 显式解锁。
5. **Combo 优先**：能用 Stand→Wave→Bow 组合表达的，禁止再写一套硬编码角度动画冒充新原子。
6. **每个新动作必须带**：speed、interrupt、pause、resume、进入检查、结束姿态、日志 MotionID。
7. **验收句式**：PR 描述必须包含「三米外可读意图：……；儿童是否默认：……；中断后姿态：……」。
